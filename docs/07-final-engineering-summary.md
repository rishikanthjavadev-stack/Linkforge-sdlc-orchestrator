# Final Engineering Summary

**Project:** LinkForge (URL shortener) + SDLC Orchestrator
**Author:** Rishikanth Deva
**Duration:** 3 phases
**Date:** July 2026

---

## 1. Plan and Rationale

### Why this scope split
The assignment asks for two distinct things graded very differently: a
working product (URL shortener) and an orchestration layer demonstrating
agentic SDLC governance. The product is well-understood, low-risk
engineering; the orchestration layer is the actual differentiator per the
evaluation criteria. The plan was structured around getting the product
out of the way fast (Phase 1) so the remaining phases could go toward the
orchestration layer's depth (Phase 2) and toward proving it actually works
rather than just describing it (Phase 3).

| Phase | Focus | Why |
|---|---|---|
| 1 | LinkForge core + requirement normalization for all 3 scenarios | Cheapest possible real target for the orchestrator to act on; normalization docs written *before* code so ambiguity-handling wasn't retrofitted |
| 2 | Orchestrator engine + all 3 scenarios run live | The graded differentiator; built to act on the real Phase 1 repo, not a mock |
| 3 | Verification, evidence capture, this summary | Turning "it should work" into "here is proof it worked, including the bugs we found" |

### Why a hand-rolled DAG executor instead of Temporal/LangGraph
Evaluated both. Temporal gives production-grade durable execution but its
setup/learning curve wasn't justified for this exercise's scope, and using it
would have meant demonstrating *Temporal's* design rather than my own.
LangGraph is a strong fit conceptually but would have split the stack
across Python and Java for no real benefit. A ~500-line custom
`WorkflowEngine` in the same Java/Spring stack as the product let every
design decision (state machine, retry/backoff, approval gating, rollback)
be fully authored, understood, and defensible — which is the point of the
exercise.

### Why the graph shape is identical across all 3 scenarios
`requirements → design → implementation → [testing, documentation] →
release` is the same DAG for greenfield, brownfield, and ambiguous. What
differs is task *behavior* (see `GraphFactory`/`StageTask` docs), not
topology. This was a deliberate choice: it proves the orchestration layer
generalizes across requirement types rather than being three special-cased
scripts wearing an orchestrator's clothing.

## 3. Risks, Trade-offs, and Validation

### Risks and trade-offs accepted (named explicitly, not hidden)
| Area | Trade-off | Why accepted |
|---|---|---|
| Short code generation | Base62(auto-increment id) is sequentially enumerable | Zero collision-handling complexity; enumerability is a known, documented risk for a prototype, not a production concern that was ignored |
| Rate limiting | In-memory per-IP token bucket, not distributed | Correct for single-instance prototype; a Redis-backed limiter is the named production follow-up |
| Analytics | Denormalized click count, no per-event log | O(1) reads; per-event analytics is a materially larger scope, explicitly deferred rather than half-built |
| Testing stage fallback | Falls back to static file-existence checks when `mvn test` can't run | A weaker guarantee than a real test pass — stated in the result message itself, not silently substituted |
| MTTR metric | A proxy (mean duration of nodes needing >1 attempt), not true incident-recovery MTTR | This prototype doesn't model a running production incident; documented as a proxy rather than mislabeled as the real thing |

### Validation approach
Every claim in this submission is backed by something that was actually
run, not just written:
- LinkForge: verified via `mvn test` (2 test classes) and manual curl
  round-trips (shorten → redirect → analytics) with real HTTP responses
  captured
- Orchestrator: verified via `mvn test` (4 test classes, 14 tests covering
  graph shape, path guardrail allow/block, change-control policy allow/block,
  and all 3 scenario behaviors against an isolated temp repo) and 3 live runs
  against the actual LinkForge repo with full request/response evidence
  captured in `docs/06-scenario-evidence.md`

### Real defects found and fixed during this process (the strongest
validation evidence, precisely because they weren't anticipated)
1. **NOT NULL constraint violation on `ShortUrl.code`** — the two-phase
   write (persist to get an id, then persist again with the derived
   code) violated the column's own NOT NULL constraint on the first
   insert. Found via a real `curl` request returning 500, root-caused via
   the actual H2 stack trace, fixed by relaxing the column constraint,
   re-verified via the same curl request returning 201.
2. **Missing `java.nio.file.Files` import in `PathGuardrail`** — introduced
   while fixing a *different* bug (macOS `/tmp` → `/private/tmp` symlink
   handling), missed because my own sanity check only verified brace
   balance, not actual symbol resolution. Found via a real `mvn compile`
   failure on the user's machine, fixed, re-verified via `mvn test`
   passing (9/9) on that same machine.
3. **Sandbox constraint, not a code defect, but load-bearing for the
   design**: this development environment cannot reach Maven Central
   (`repo.maven.apache.org` returns 403 through the network proxy). This
   is precisely the condition `TestingTask`'s fallback path is designed to
   handle — the fallback logic was validated by a real environmental
   constraint, not a contrived test double.

## 4. Assumptions

- The orchestrator and LinkForge are run as sibling directories on the
  same machine, with the orchestrator granted filesystem access to write
  into LinkForge's source tree. In a real organization this would need an
  actual authorization/permissions model (e.g. a bot identity with scoped
  repo write access, or a PR-based workflow instead of direct writes) —
  out of scope for a prototype but named here rather than glossed over.
- "Approval" and "clarification" are modeled as trusting whatever note
  text is passed to the API; there is no authentication layer identifying
  *which* human approved something. A production version would need this
  tied to real identity/SSO.
- The three required scenarios are treated as independent demo runs
  rather than a single continuous project history. A more mature version
  would model how a brownfield change built on top of an already-released
  greenfield workflow, carrying context forward across releases.

## 5. Limitations

- **No true distributed execution**: the orchestrator runs as a single
  Spring Boot process with an in-memory `ConcurrentHashMap` of workflow
  instances. Restarting the orchestrator loses all workflow state. A
  production version would persist workflow/node state to a database.
- **No horizontal scaling of the rate limiter it generates**: documented
  in `docs/03-brownfield-requirement-normalization.md` as a named,
  intentional trade-off.
- **MTTR is a proxy metric**, not a true production incident-recovery
  measurement (see trade-offs table above).
- **Policy guardrails cover two concerns, not a complete policy engine**:
  `PathGuardrail` (containment - is the write inside the repo) and
  `ChangeControlPolicy` (change management - does this path require
  explicit human authorization even inside the repo, e.g. `pom.xml` and
  the domain layer). Both are real, unit-tested, load-bearing checks. A
  production system would need broader coverage still (dependency
  allowlisting, secret-scanning before commit, static analysis gates) -
  these two demonstrate the *pattern* of enforced, code-level guardrails
  spanning both a security concern and a change-control concern, rather
  than a single check standing in for both.
- **Re-planning is demonstrated only for not-yet-started downstream
  nodes** (the ambiguous scenario). The state machine supports resetting
  any node back to `PENDING`, so invalidating *already-completed*
  downstream work on a later upstream change is a straightforward
  extension, but it isn't exercised by the current 3 scenarios since none
  of them change a requirement after implementation has already run.
